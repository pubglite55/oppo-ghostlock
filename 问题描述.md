### 1. GhostLock 触发 ✅

```
FUTEX_LOCK_PI target: ret=0 ✅

时序要求: waiter 调用 `FUTEX_WAIT_REQUEUE_PI` 后需要等待约 3 秒才能真正阻塞在内核中。

### 3. configfs_bin_file_operations 找到 ✅

```
configfs_bin_file_operations @ 0xffffffc00a175920
  +0x10: read_iter   = configfs_read_bin_file
  +0x18: write_iter  = configfs_write_bin_file

`configfs_read_bin_file` 使用 `file->private_data + 88` 作为 `bin_buffer` 指针。

```
ashmem_area.name[88] ↔ configfs_buffer.bin_buffer

写入 ashmem name → 控制 bin_buffer 指针 → 任意内核读写
```

## 当前阻塞点

### 核心问题: 如何将 ashmem_misc.fops 覆盖为 configfs_bin_file_operations?

exploit 原始流程:
```
1. KernelSnitch 泄漏 mm_struct 地址
   → 计算 SLUB slab 基址 (order-3, 32KB)
   ↓
2. sendmsg 发送 SKB payload
   → 在 slab 中布置 fake kernel page
   → fake_page 包含 fake_lock → 指向 ashmem_misc.fops
   ↓
3. FUTEX_CMP_REQUEUE_PI 触发 GhostLock
   → dangling pi_blocked_on 指针
   ↓
4. pselect 操纵内核栈
   → 内核跟随 fake_lock 写入 ashmem_misc.fops
   → ashmem_misc.fops = configfs_bin_file_operations
   ↓
5. 打开 /dev/ashmem → 类型混淆 → 任意读写 → 提权
```

**阻塞在第 1-2 步**: 没有 mm_struct 地址，无法布置 fake kernel page。

### 为什么需要 mm_struct

`prepare_skb_payload(base)` 需要 mm_struct 地址来:
1. 计算 `base = leaked & ~(ORDER3_SIZE - 1)` (SLUB slab 基址)
2. 在 slab 中布置 fake kernel page
3. fake page 包含 fake_lock → 指向 ashmem_misc.fops

## 已尝试的 mm_struct 泄漏方法

| 方法 | 结果 | 原因 |
|------|------|------|
| KernelSnitch (futex hash timing) | ❌ | KPTI 启用，cntvct_el0 精度不足 |
| perf_event_open | ❌ | SELinux 阻止 (errno=13) |
| /proc/self/pagemap | ❌ | 全零 |
| /proc/kallsyms | ❌ | kptr_restrict=2 |
| sendmsg + cmsg | ❌ | 只有用户态数据 |
| io_uring | ❌ | SQ/CQ 中无内核指针 |
| BPF_MAP_CREATE | ❌ | EPERM |
| userfaultfd | ❌ | EPERM |

## 核心问题

**如何在 KPTI + SELinux Enforcing + kptr_restrict=2 的 Android 设备上，获取内核地址泄漏（mm_struct 或其他内核符号地址）？**

或者更具体地: **如何在不依赖 KernelSnitch 时序侧信道的情况下，找到 mm_struct 地址或直接覆盖 ashmem_misc.fops？**

### 可能的方向

1. **替代 mm_struct 泄漏方法** — 不依赖时序的内核地址泄漏
2. **直接覆盖 ashmem_misc.fops** — 绕过 mm_struct，找到其他方式写入 fops
3. **栈覆盖替代方案** — 不用 pselect，用其他 syscall 将用户数据写入 waiter 位置
4. **其他漏洞利用** — 结合不同漏洞实现内核地址泄漏
